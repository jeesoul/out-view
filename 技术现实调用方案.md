# outView 系统架构设计方案 (基于 Spring Boot)

## 1. 技术选型可行性分析

### 1.1 核心结论
**完全可行，且是成熟的企业级方案。**
使用 **Java + Spring Boot + Netty** 构建 `outView` 服务端是业界非常标准的做法。
- **Spring Boot**: 负责应用的生命周期管理、配置加载、依赖注入、日志监控、REST API（用于管理后台）等“非网络核心”业务。
- **Netty**: 作为 Spring Boot 内部的嵌入式高性能网络引擎，专门处理高并发的 TCP/UDP 连接、数据编解码、流量转发和心跳保活。

### 1.2 为什么选择这个组合？
| 维度         | Spring Boot + Netty 优势                                     | 对比其他方案 (如 Go/FRP)                                     |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **开发效率** | Spring 生态极其丰富，快速搭建管理后台、鉴权逻辑、数据库交互。 | Go 语言虽然性能略高，但在复杂业务逻辑（如用户管理、计费、审计）上 Java 生态更完善。 |
| **性能表现** | Netty 基于 NIO (Reactor 模式)，单线程可处理数万并发连接，完全满足远程桌面场景。 | 性能略低于纯 C/Go 实现，但对于远程桌面（带宽瓶颈通常大于 CPU 瓶颈）完全够用。 |
| **稳定性**   | Netty 经过阿里、腾讯等大厂多年验证，稳定性极高。             | 相当。                                                       |
| **人才储备** | Java 开发人员众多，便于后续维护和招聘。                      | Go 相对小众一些。                                            |
| **跨平台**   | "Write Once, Run Anywhere"，服务端可部署在 Linux/Windows/Mac。 | 相当。                                                       |

## 2. 总体架构设计

### 2.1 逻辑架构图
```mermaid
graph TD
    User[用户 (控制端)] -->|RDP 协议 | EntryPoint[入口端口 (Netty)]
    Client[家庭电脑 (outView Client)] -->|长连接隧道 | ControlPlane[控制平面 (Netty)]
    
    subgraph "outView Server (Spring Boot Application)"
        EntryPoint -->|路由转发 | TunnelManager[隧道管理器]
        ControlPlane -->|注册/心跳 | TunnelManager
        TunnelManager -->|绑定关系 | SessionStore[会话存储 (Redis/内存)]
        
        SpringCore[Spring 核心容器] -->|管理 | NettyServer
        SpringCore -->|管理 | AuthService[鉴权服务]
        SpringCore -->|管理 | ConfigService[配置服务]
        
        REST_API[REST API 控制器] -->|Web 管理台 | Admin[管理员]
    end
    
    TunnelManager -.->|数据透传 (零拷贝)| EntryPoint
```

### 2.2 核心组件职责

1.  **Spring Boot 容器层**:
    *   负责启动 Netty Server。
    *   提供 HTTP REST API 供管理后台调用（查看在线设备、重置 Token、查看日志）。
    *   集成 MyBatis/JPA 进行持久化存储（用户信息、设备绑定关系）。
    *   集成 Redis 存储实时会话状态（可选，单机可用内存）。

2.  **Netty 网络层 (核心)**:
    *   **Boss Group**: 负责接收客户端连接请求 (Accept)。
    *   **Worker Group**: 负责处理 IO 读写 (Read/Write)。
    *   **Control Channel Handler**: 处理客户端注册、心跳、鉴权指令。
    *   **Data Channel Handler**: 处理实际的业务数据转发（RDP 流量），采用**零拷贝 (Zero-Copy)** 技术提升性能。

3.  **客户端 (outView Client)**:
    *   *建议*: 虽然服务端用 Java，但客户端建议仍使用 **Go** 或 **Rust** 编译为单一二进制文件，方便用户下载运行，无 JRE 依赖困扰。如果必须全栈 Java，则需打包 JRE (体积较大，约 100MB+)。

## 3. 详细模块设计

### 3.1 通信协议设计 (自定义二进制协议)
为了区分控制指令和数据流量，需要在 TCP 流之上定义简单的应用层协议。

**消息头结构 (Header - 12 Bytes):**
| 字段         | 长度 (Byte) | 说明                                              |
| :----------- | :---------- | :------------------------------------------------ |
| Magic Number | 4           | 固定标识 `0x4F565753` ("OVWS")，防止非法连接      |
| Version      | 1           | 协议版本                                          |
| Type         | 1           | 消息类型 (1: 注册, 2: 心跳, 3: 数据转发, 4: 错误) |
| Length       | 4           | _body_ 长度                                       |
| Reserved     | 2           | 保留字段                                          |

**消息体 (Body):**
*   **注册包**: `{ "deviceId": "uuid", "token": "secret", "localPort": 3389 }` (JSON)
*   **心跳包**: `{ "timestamp": 1710000000 }`
*   **数据包**: 原始二进制流 (RDP 数据)，直接透传，不解析内容。

### 3.2 核心流程设计

#### A. 客户端上线流程
1.  Client 启动，读取配置文件中的 Server IP 和 Token。
2.  Client 连接 Server 的 **控制端口** (例如 7000)。
3.  Client 发送 **注册包**。
4.  Server (`AuthHandler`) 校验 Token 合法性。
5.  Server 校验通过后，将该连接标记为 `Active`，并记录 `DeviceID <-> Channel` 映射关系。
6.  Server 返回 **注册成功** 响应，并告知分配的外部访问端口 (例如 6001)。
7.  Client 进入心跳循环 (每 30s 发送一次)。

#### B. 外部用户连接流程 (RDP)
1.  用户在外出电脑打开 MSTSC，输入 `ServerIP:6001`。
2.  请求到达 Server 的 **数据端口池** (6000-6500)。
3.  Server (`ProxyHandler`) 根据目标端口 `6001` 查找映射表，找到对应的 `DeviceID` 和 `Client Channel`。
4.  Server 同时建立两条通道：
    *   `User Channel` (与外出电脑)
    *   `Client Channel` (与家庭电脑，已存在)
5.  Server 启动双向数据转发任务 (`RelayTask`)，利用 `Channel.read()` 和 `Channel.write()` 将数据在两个 Channel 间无缝搬运。

**Netty 服务启动类 (Spring Bean):**
```java
@Component
public class NettyServer {
    @Value("${netty.port}")
    private int port;

    @PostConstruct
    public void start() throws InterruptedException {
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        EventLoopGroup workerGroup = new NioEventLoopGroup();

        try {
            ServerBootstrap b = new ServerBootstrap();
            b.group(bossGroup, workerGroup)
             .channel(NioServerSocketChannel.class)
             .childHandler(new ChannelInitializer<SocketChannel>() {
                 @Override
                 protected void initChannel(SocketChannel ch) {
                     ch.pipeline().addLast(new FrameDecoder()); // 拆包粘包处理
                     ch.pipeline().addLast(new AuthHandler());  // 鉴权
                     ch.pipeline().addLast(new HeartbeatHandler());// 心跳
                     ch.pipeline().addLast(new ProxyHandler()); // 数据转发核心
                 }
             });
            
            // 开启 TCP 底层优化
            b.childOption(ChannelOption.TCP_NODELAY, true);
            b.childOption(ChannelOption.SO_KEEPALIVE, true);

            ChannelFuture f = b.bind(port).sync();
            System.out.println("OutView Server started on port: " + port);
            f.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}
```

**数据转发核心逻辑 (ProxyHandler 简化版):**
```java
// 当外部用户连接进来时
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    // 1. 识别当前连接的目标端口，找到对应的内网 Client Channel
    String deviceId = portMappingService.getDeviceByPort(ctx.channel().localAddress().getPort());
    Channel clientChannel = sessionStore.getClientChannel(deviceId);

    if (clientChannel != null && clientChannel.isActive()) {
        // 2. 将数据直接写入 Client Channel (零拷贝思想)
        clientChannel.writeAndFlush(msg);
    } else {
        // 3. 客户端离线，关闭连接
        ctx.close();
    }
}
```

## 4. 性能优化策略

针对远程桌面场景（大流量、低延迟要求），需在架构设计中强调以下优化：

1.  **零拷贝 (Zero-Copy)**:
    *   在 Netty 中，避免将 `ByteBuf` 转换为 Java 字节数组。直接使用 `FileRegion` 或在 Channel 间直接传递 `ByteBuf` 引用，减少 CPU 上下文切换和内存复制。
2.  **内存池化 (Memory Pooling)**:
    *   启用 Netty 的 `PooledByteBufAllocator`，复用内存缓冲区，减少 GC 压力。
3.  **Epoll 原生支持**:
    *   在 Linux 环境下，强制使用 `EpollEventLoopGroup` 代替 `NioEventLoopGroup`，利用 Linux 内核特性提升 IO 效率。
4.  **参数调优**:
    *   调整 JVM 参数：`-Xms`, `-Xmx` 设为一致，使用 G1 垃圾收集器。
    *   操作系统层面：调整文件描述符限制 (`ulimit -n 65535`)，调整 TCP 内核参数 (`tcp_tw_reuse`, `somaxconn`)。

## 5. 安全架构设计

1.  **双向认证**:
    *   客户端连接时必须携带高强度 Token。
    *   服务端验证 Token 后才允许建立隧道。
2.  **传输加密**:
    *   **方案 A (推荐)**: 在 Netty Pipeline 中加入 `SslHandler`。服务端配置 SSL 证书，客户端信任该证书。所有转发数据自动加密。
    *   **方案 B**: 依赖 RDP 自带的加密（RDP 本身支持 TLS），Server 仅做盲转发。*风险：Server 可能被中间人攻击，建议采用方案 A。*
3.  **端口隔离**:
    *   每个租户/设备分配独立的对外端口，防止端口扫描撞库。
    *   防火墙策略：仅开放配置的端口范围，拒绝其他所有入站请求。
4.  **防 DDoS**:
    *   在 Netty 入口处限制单 IP 的连接速率 (`ChannelHandler` 中实现简单的令牌桶算法)。

## 6. 部署架构

### 6.1 单机部署 (MVP 阶段)
*   **OS**: Ubuntu 22.04 LTS
*   **Runtime**: JDK 17+ (推荐使用 GraalVM 编译为 Native Image 以减小体积和加快启动，可选)
*   **Process**: 使用 `systemd` 管理 Spring Boot Jar 包。
*   **Config**: 外部化 `application.yml`。

### 6.2 集群部署 (未来扩展)
如果需要支持数千个并发连接：
*   **负载均衡**: 使用 Nginx 或 LVS 在多台 OutView Server 前做 TCP 负载均衡 (基于端口段划分，如 Server A 负责 6000-6200, Server B 负责 6201-6400)。
*   **状态共享**: 使用 Redis 集群存储 `DeviceID <-> ServerIP:Port` 的映射关系，实现无状态化（稍微复杂，因为 TCP 长连接是有状态的，通常采用**端口段静态分片**策略更简单有效）。

## 8. 给架构师的特别提示 (Risk & Tips)

1.  **粘包/拆包问题**: 必须在 Netty Pipeline 中正确配置 `LengthFieldBasedFrameDecoder`，否则长时间运行后会出现数据错乱。
2.  **资源泄露**: 务必在 `channelInactive` 回调中清理所有关联的资源（Map 中的引用、定时任务等），否则长时间运行会导致 OOM。
3.  **JDK 版本**: 强烈建议使用 **JDK 17** 或 **JDK 21** (LTS 版本)，利用虚拟线程 (Project Loom) 可能在未来进一步简化并发模型，但目前 Netty 的 Reactor 模型已足够优秀。
4.  **客户端语言**: 再次建议**客户端不要用 Java**。Java 客户端需要用户安装 JRE 或打包巨大的 Runtime，体验不好。建议客户端用 **Go** 编写，编译成几十 MB 的单文件，用户体验更佳。服务端用 Java Spring Boot 完全没问题。

这份方案可以直接交给架构师进行详细设计和代码落地。
